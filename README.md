# ArchVisionAI

**ArchVisionAI** – інтелектуальна вебсистема для аналізу та класифікації архітектурних стилів за зображенням будівлі.

Проєкт поєднує моделі комп’ютерного зору, генеративний ШІ для пояснення результату, інтерактивну карту, історію аналізів користувача та вебінтерфейс для зручної роботи із завантаженими зображеннями.

## Демо

Розгорнута версія вебзастосунку доступна за посиланням:

http://archvisionai.swedencentral.cloudapp.azure.com

## Призначення проєкту

Мета ArchVisionAI – не лише визначити архітектурний стиль будівлі на зображенні, а й подати результат у зрозумілому для користувача вигляді.

Після завантаження зображення система:

- визначає архітектурний стиль будівлі;
- показує рівень впевненості моделі;
- відображає кілька найімовірніших варіантів класифікації;
- формує розширений текстовий опис за допомогою Gemini API;
- показує географічний контекст визначеного стилю на інтерактивній карті;
- зберігає історію аналізів для авторизованих користувачів;
- формує статистику та персональні архітектурні вподобання користувача.

Такий підхід робить застосунок корисним як навчальний, демонстраційний і довідковий інструмент для роботи з архітектурними зображеннями, культурною спадщиною, туристичними матеріалами та задачами комп’ютерного зору.

## Основні можливості

### Класифікація архітектурного стилю

Користувач може завантажити зображення будівлі через вебінтерфейс. Після цього система виконує попередню обробку зображення, передає його до моделі комп’ютерного зору та повертає прогнозований архітектурний стиль.

Класифікація виконується за **25 архітектурними стилями**.

### Підтримка кількох режимів аналізу

У застосунку доступні такі режими:

- **EfficientNet-B0** – компактна та швидка згорткова нейронна мережа;
- **ResNet-50** – глибша CNN-архітектура із залишковими зв’язками;
- **Ensemble** – комбінування результатів EfficientNet-B0 і ResNet-50;
- **TTA** – test-time augmentation, тобто аугментація зображення на етапі аналізу для підвищення стабільності результату.

### Розширений опис Gemini AI

Окрім числового результату класифікації, система може сформувати пояснення за допомогою Gemini API.

Розширений опис може містити:

- визначений архітектурний стиль;
- можливу назву будівлі;
- місцезнаходження;
- архітектора;
- рік або період будівництва;
- характерні архітектурні елементи;
- історичну цінність;
- сучасне використання будівлі.

### Інтерактивна карта

Для кожного архітектурного стилю в проєкті підготовлено географічні метадані. Вони використовуються для відображення регіонів поширення стилю та прикладів відомих будівель.

Карта реалізована за допомогою **Leaflet** і підтримує:

- маркери відомих будівель;
- географічні області поширення стилів;
- інформаційні вікна з коротким описом;
- візуальне доповнення результату класифікації.

### Автентифікація користувача

Незареєстрований користувач може виконувати базовий аналіз зображення. Після реєстрації та входу в систему відкриваються додаткові можливості:

- збереження історії аналізів;
- перегляд попередніх результатів;
- збереження мініатюр зображень;
- статистика за кількістю аналізів;
- найчастіше визначені стилі;
- персональні архітектурні вподобання.

### Збереження даних

Для збереження даних використовується **SQLite**. У базі даних зберігаються:

- користувачі;
- хеші паролів і salt-значення;
- історія аналізів;
- мініатюри зображень;
- визначені архітектурні стилі;
- значення впевненості моделі;
- результати Gemini-аналізу;
- архітектурні вподобання користувача.

## Архітектура застосунку

Проєкт побудовано за багатошаровою архітектурою:

1. **Клієнтський рівень**  
   Користувач відкриває вебінтерфейс, вибирає модель, завантажує зображення та переглядає результат.

2. **Серверний рівень**  
   Flask приймає HTTP-запити, перевіряє дані, керує сесіями та повертає JSON-відповіді.

3. **Сервісний рівень**  
   `AnalysisService` готує зображення, створює мініатюру, визначає режим аналізу та викликає ML-аналізатор.

4. **Рівень машинного навчання**  
   `ArchVisionAnalyzer` завантажує моделі, виконує попередню обробку зображення, запускає інференс, застосовує softmax і формує прогноз.

5. **Рівень Gemini AI**  
   Gemini API формує розширений україномовний опис архітектурного об’єкта.

6. **Рівень бази даних**  
   SQLite зберігає користувачів, історію аналізів, статистику та архітектурні вподобання.

7. **Рівень геопросторової візуалізації**  
   Leaflet відображає регіони поширення стилю та приклади відомих будівель на карті.

## Режими моделей

### EfficientNet-B0

EfficientNet-B0 використовується як компактна й швидка CNN-модель для базової класифікації архітектурних стилів. Вона має вхідний розмір зображення 224×224 пікселі та 25 вихідних класів.

### ResNet-50

ResNet-50 є глибшою згортковою нейронною мережею із залишковими зв’язками. Вона використовується як стабільна модель для класифікації зображень архітектурних фасадів.

### Ensemble

У режимі Ensemble система усереднює ймовірності, отримані від EfficientNet-B0 і ResNet-50. Це зменшує вплив помилок окремої моделі та робить підсумковий прогноз стабільнішим.

### Test-Time Augmentation

Якщо увімкнено TTA, система створює п’ять варіантів одного зображення:

- оригінальне зображення;
- дзеркальне відображення;
- зображення з підвищеною яскравістю;
- зображення з підвищеним контрастом;
- зображення з невеликим поворотом.

Після цього модель аналізує всі варіанти, а фінальні ймовірності усереднюються.

## Змінні середовища

Перед запуском потрібно створити файл `.env` у корені проєкту.

Приклад:

```env
GEMINI_MODEL=gemini-2.5-flash
GEMINI_API_KEY_1=
GEMINI_API_KEY_2=
GEMINI_API_KEY_3=
SECRET_KEY=
```

`SECRET_KEY` потрібен для роботи Flask-сесій.

Ключі Gemini API потрібні для формування розширеного ШІ-опису. Якщо ключі не задані або ліміт Gemini вичерпано, базова класифікація зображень власними моделями все одно може працювати.

## Файли моделей

Для роботи класифікації потрібні навчені checkpoint-файли моделей. Вони очікуються у директорії:

```text
checkpoints/
```

Очікувані файли:

```text
checkpoints/efficientnet_b0_best.pth
checkpoints/resnet50_best.pth
```

Якщо checkpoint-файли відсутні, вебзастосунок може запуститися, але класифікація відповідними моделями буде недоступною.

## Локальний запуск

### 1. Клонувати репозиторій

```bash
git clone https://github.com/yana-donda/ArchVisionAI.git
cd ArchVisionAI
```

### 2. Створити віртуальне середовище

```bash
python -m venv .venv
```

Активувати середовище:

```bash
# Windows
.venv\Scripts\activate
```

```bash
# Linux / macOS
source .venv/bin/activate
```

### 3. Встановити залежності

```bash
pip install --upgrade pip
pip install -r requirements.txt
```

### 4. Створити `.env`

```bash
cp .env.example .env
```

Після цього потрібно заповнити `SECRET_KEY` і, за потреби, Gemini API keys.

### 5. Підготувати датасет

Для навчання моделей потрібно завантажити датасет **Architectural styles** з Kaggle:

```text
https://www.kaggle.com/datasets/dumitrux/architectural-styles-dataset
```

Після завантаження дані потрібно розмістити у форматі `ImageFolder`, тобто кожен архітектурний стиль має бути окремою папкою з відповідними зображеннями.

Очікувана структура датасету:

```text
dataset/
├── train/
│   ├── Achaemenid architecture/
│   ├── American Foursquare architecture/
│   ├── American craftsman style/
│   └── ...
├── val/
│   ├── Achaemenid architecture/
│   ├── American Foursquare architecture/
│   ├── American craftsman style/
│   └── ...
└── test/
    ├── Achaemenid architecture/
    ├── American Foursquare architecture/
    ├── American craftsman style/
    └── ...
```

Папка `train` використовується для навчання моделей, `val` – для перевірки якості під час навчання та вибору найкращих ваг, а `test` – для фінального оцінювання моделі після завершення навчання.

Назви папок класів у `train`, `val` і `test` мають збігатися, щоб модель працювала з однаковим набором архітектурних стилів на всіх етапах.

### 6. Навчити моделі та отримати checkpoint-файли

У проєкті передбачені окремі скрипти для навчання моделей **EfficientNet-B0** і **ResNet-50**.

Для навчання EfficientNet-B0 потрібно виконати:

```bash
python train_efficientnet_b0.py
```

Для навчання ResNet-50 потрібно виконати:

```bash
python train_resnet50.py
```

За замовчуванням ці скрипти використовують такі шляхи до датасету:

```text
dataset/train
dataset/val
```

Після запуску навчання автоматично створюється директорія:

```text
checkpoints/
```

У цій директорії зберігаються файли з найкращими вагами моделей:

```text
checkpoints/efficientnet_b0_best.pth
checkpoints/resnet50_best.pth
```

Саме ці checkpoint-файли використовуються вебзастосунком для класифікації архітектурних стилів.

Також під час навчання створюється директорія `training_runs`, у якій зберігаються логи, метрики та короткі підсумки навчання.

Якщо потрібно явно вказати шляхи до датасету або змінити кількість епох, скрипти можна запускати з параметрами:

```bash
python train_efficientnet_b0.py --data-dir dataset/train --val-dir dataset/val --epochs 20
```

```bash
python train_resnet50.py --data-dir dataset/train --val-dir dataset/val --epochs 20
```

За потреби можна додати параметр `--pretrained`, щоб використати попередньо навчені ImageNet-ваги:

```bash
python train_efficientnet_b0.py --pretrained
```

```bash
python train_resnet50.py --pretrained
```

Якщо checkpoint-файли вже були отримані раніше, повторно навчати моделі не потрібно. Достатньо просто помістити файли `efficientnet_b0_best.pth` і `resnet50_best.pth` у директорію `checkpoints`.

Локально вебзастосунок буде доступний за адресою:

## Тестування моделей

Під час тестування було перевірено такі режими:

- EfficientNet-B0;
- ResNet-50;
- Ensemble;
- EfficientNet-B0 + TTA;
- ResNet-50 + TTA;
- Ensemble + TTA.

Узагальнені результати:

| Модель | Точність | F1-macro | Час аналізу |
|---|---:|---:|---:|
| EfficientNet-B0 | 86.13% | 0.8583 | 0.0442 с |
| ResNet-50 | 87.21% | 0.8709 | 0.0476 с |
| Ensemble | 88.44% | 0.8824 | 0.0942 с |
| EfficientNet-B0 + TTA | 87.48% | 0.8718 | 0.303 с |
| ResNet-50 + TTA | 88.05% | 0.8796 | 0.312 с |
| Ensemble + TTA | 89.73% | 0.8952 | 0.6451 с |

Найкращу якість класифікації показав режим **Ensemble + TTA**.

## Безпека

У застосунку реалізовано базові механізми безпеки:

- паролі не зберігаються у відкритому вигляді;
- для паролів використовується PBKDF2-HMAC-SHA256;
- для кожного користувача генерується окреме salt-значення;
- обмежено максимальний розмір вхідного файла;
- секретні ключі мають зберігатися у `.env`;
- `.env` не слід додавати до GitHub-репозиторію.

## Обмеження

Поточна версія має кілька практичних обмежень:

- якість результату залежить від ракурсу, освітлення та якості зображення;
- візуально схожі архітектурні стилі можуть плутатися між собою;
- Gemini API має власні ліміти запитів;
- SQLite підходить для демонстраційного або невеликого застосунку, але для більшого навантаження краще перейти на серверну СКБД.

## Можливі напрями розвитку

У майбутньому проєкт можна розширити такими покращеннями:

- додати більше архітектурних стилів;
- збалансувати навчальний датасет;
- додати більше зображень у різних умовах освітлення, масштабу та ракурсу;
- покращити класифікацію візуально схожих стилів;
- показувати користувачу кілька найімовірніших стилів з поясненням;
- додати сегментацію архітектурних елементів;
- інтегрувати онлайн-каталоги архітектурних пам’яток;
- розширити базу географічних регіонів і відомих будівель;
- покращити персоналізовані рекомендації на основі історії аналізів;
- реалізувати англомовний інтерфейс або повну багатомовну підтримку для користувачів з інших країн;
- перейти на PostgreSQL або іншу серверну базу даних для production-сценаріїв.
